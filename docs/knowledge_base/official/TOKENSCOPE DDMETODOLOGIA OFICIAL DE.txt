Versão: 1.0Status: Documento operacional oficialFinalidade: orientar, de forma permanente, como o projeto TokenScope DD deve ser planejado, executado, auditado, aprovado e versionado.

======================================================================

PRINCÍPIO CENTRAL======================================================================

O TokenScope DD será desenvolvido por uma equipe assistida por inteligência artificial, mas governada pelo humano.

A divisão oficial é:

O FUNDADOR define direção, prioridades, limites, riscos e aprovação final.

O CHATGPT atua como diretor de produto, arquiteto, planejador e auditor.

O CODEX atua como engenheiro executor local dentro do VS Code.

O GIT E O GITHUB preservam histórico, versões e pontos de recuperação.

Princípio operacional:

O humano define direção e limites.O ChatGPT amplia análise, arquitetura e supervisão.O Codex executa, testa e produz evidências.O Git registra somente o que foi aprovado.

Princípio de produto:

A IA explica e auxilia a consulta.O sistema calcula.As evidências sustentam.O humano decide.

2.1 FUNDADOR DO PROJETO

O fundador é a autoridade final do TokenScope DD.

Responsabilidades:

definir a visão;

definir o problema que o produto deve resolver;

definir prioridades;

decidir o que é público ou privado;

aprovar mudanças relevantes;

aprovar custos e serviços pagos;

aprovar publicação;

aprovar metodologia e regras de negócio;

aceitar, adiar ou rejeitar riscos;

decidir quando uma entrega tem valor;

dar a palavra final.

O fundador não precisa competir tecnicamente com as inteligências artificiais.

Ele não precisa revisar sozinho:

cada linha de Python;

imports;

estrutura interna de testes;

configuração de ambiente;

detalhes de empacotamento;

todos os comandos Git;

decisões técnicas pequenas e reversíveis.

O fundador deve fiscalizar principalmente:

Utilidade:Isso realmente ajuda alguém a analisar um ativo?

Coerência:Isso representa o que foi definido para o TokenScope?

Compreensão:O resultado pode ser entendido sem depender de um programador?

Risco:Algo está sendo apagado, publicado, cobrado ou afirmado sem segurança?

Prioridade:Esta é realmente a próxima tarefa mais importante?

Regra padrão:

UMA TAREFA PRINCIPAL DE IMPLEMENTAÇÃO POR VEZ.

Trabalhos paralelos só podem ocorrer quando:

são independentes;

não alteram os mesmos arquivos;

não disputam o mesmo contrato;

não criam estados conflitantes.

Exemplo seguro:

um agente audita schemas sem alterar arquivos;

outro agente documenta conceitos sem alterar o mesmo documento.

Exemplo inseguro:

dois agentes alteram config.py ao mesmo tempo;

dois agentes modificam o mesmo pipeline;

uma tarefa muda contrato enquanto outra o implementa.

Mudanças complexas não devem ser executadas em uma única tarefa.

Exemplo para scoring:

TS-020 — Definição das dimensõesTS-021 — Definição das featuresTS-022 — Política de dados ausentesTS-023 — NormalizaçãoTS-024 — Pesos e penalidadesTS-025 — Motor determinísticoTS-026 — Testes de coerênciaTS-027 — ExplicaçãoTS-028 — Validação com dados reais

Cada tarefa deve atravessar os gates antes da seguinte.

Não combinar automaticamente:

alteração + commit + push

Fluxo preferido:

Codex altera.

Codex testa.

Codex relata.

ChatGPT audita.

Fundador aprova.
